# 进阶任务：多候选生成与 NLI 重排序的事实纠错策略

## 任务选择

本项目进阶任务选择“纠错方面”：设计一种比直接局部重写更稳定的纠错策略，并说明该方法为何适合长文摘要场景。

基础纠错流程是：

```text
检测到错误句 -> Qwen 直接改写 -> 替换原句
```

这种方法的问题是，生成模型只输出一个纠错句，一旦该句复制证据、输出过长、包含对话模板，或仍然不被原文证据支持，纠错就会失败。

因此，我们将纠错策略改为：

```text
检测到错误句
-> Qwen 生成多个候选纠错句
-> 对每个候选做格式过滤
-> 对格式合格候选做 NLI 评分
-> 选择 entailment 分数最高且达到一致性阈值的候选
-> 如果都不达标，则保留原句并标记未修复
```

## 方法说明

1. 多候选生成

对每个被检测为不一致的摘要句，系统不再只生成一个纠错结果，而是通过 `corrector.num_candidates` 控制候选数量。默认 full 配置中生成 3 个候选。

2. 格式过滤

每个候选句都要通过原有格式校验，包括：

- 非空输出
- 不能和原句完全相同
- 不能超过原句长度比例
- 不能包含 `Human`、`Assistant`、`User` 等对话残留
- 不能像复制了一整段证据
- 去除重复候选

3. NLI 重排序

对每个格式合格候选，使用 BART-MNLI 计算：

```text
NLI(evidence, candidate)
```

其中 `evidence` 是从原文检索到的证据片段，`candidate` 是候选纠错句。系统选择 entailment score 最高的候选。

4. 保守接受策略

只有当最佳候选达到一致性阈值，即：

```text
candidate entailment score >= 0.5
```

才替换原句。否则不强行修改摘要，保留原句，并在结果中记录：

```text
rerank_decision = no_candidate_passed_nli
```

这样可以避免把一个格式上合格、但事实上仍不被证据支持的句子写回摘要。

## 与基础纠错的区别

基础纠错：

```text
错误句 -> 单次生成 -> 格式合格就替换
```

进阶纠错：

```text
错误句 -> 多候选生成 -> 格式过滤 -> NLI 打分 -> 只接受证据支持的最佳候选
```

因此，进阶策略不再完全依赖 Qwen 的一次生成结果，而是把 NLI 作为候选选择器和事实约束器。

## 为什么适合长文摘要

长文摘要中的事实错误通常分散在少数句子中，错误类型可能包括实体错误、数字错误、范围错误和因果关系错误。如果直接重写整篇摘要，容易引入新的事实错误，也难以分析纠错模块的效果。

多候选生成与 NLI 重排序适合长文摘要，原因是：

- 仍然保持局部修改，避免重写整篇摘要带来的二次幻觉。
- 多候选增加了找到正确改写的机会。
- NLI 重排序让候选选择依赖原文证据，而不是只依赖生成模型表面流畅度。
- 如果所有候选都不被证据支持，系统选择不替换，体现保守纠错原则。

## 代码实现位置

- 配置文件：`config.yaml`
  - `corrector.num_candidates: 3`
- 纠错模块：`src/correction/corrector.py`
  - `_generate_candidate_texts`
  - `_build_candidate_records`
  - `_rerank_candidates`
  - `correct_single(..., nli_checker=...)`
- Pipeline：`src/pipeline.py`
  - 当 `corrector_num_candidates > 1` 时，在纠错阶段加载 NLI checker，并传入 corrector 做候选重排序。

## 答辩表述

可以这样介绍：

> 我们的进阶任务选择纠错方向。基础纠错是检测到错误句后直接让 Qwen 局部改写，但单次生成不稳定。我们进一步实现了“多候选生成 + NLI 重排序”的保守纠错策略：对每个风险句生成多个候选，先做格式过滤，再用 BART-MNLI 计算每个候选与原文证据之间的 entailment 分数，只接受分数最高且达到一致性阈值的候选。如果所有候选都不达标，就保留原句并标记未修复。这样可以降低单次生成失败和二次幻觉风险，更适合长文摘要的事实纠错场景。
